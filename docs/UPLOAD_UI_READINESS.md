## Readiness Summary

GRWM is ready for a narrowly scoped real wardrobe image upload UI implementation because the upload boundary now exists before the UI:

- Users can sign in, record privacy consent, complete wardrobe onboarding, and reach Wardrobe Home.
- Installed development-build wardrobe onboarding manual QA passed on 2026-06-15 using the local QA access harness without skipping consent or setup.
- Wardrobe Home clearly states that real image upload is inactive and exposes only a disabled placeholder CTA.
- `wardrobeItems` supports draft/upload lifecycle fields separately from `wardrobeSetupProfiles`.
- Firestore draft creation is field-bound and owner-bound.
- Storage upload paths are exact, private, owner-bound, metadata-bound, and rules-tested.
- Backend finalisation validates finalized objects and existing Firestore records before marking `uploaded`.
- Upload finalisation does not create AI jobs and keeps `analysisStatus: not_requested`.
- Consent gates separate private upload from AI photo analysis.
- Account deletion covers wardrobe setup records, wardrobe item records, and private wardrobe Storage prefixes.
- Orphan detection is non-destructive and documented.

## Prerequisite Matrix

### A. User Flow

- User can sign in: ready. Email/password auth, persisted auth state, protected routing, and manual QA evidence exist.
- User can complete privacy consent: ready. `privacyConsents/{userId}` is created from the mobile Privacy screen and editable from Settings.
- User can complete wardrobe onboarding: ready. Installed development-build A-J QA passed.
- User can reach Wardrobe Home: ready. The completed setup path navigates to `wardrobe`.
- Wardrobe Home upload inactive state: ready. Copy says image upload is not active yet and "Add wardrobe item soon" is disabled.
- Manual QA has passed: ready for the current pre-upload state.

### B. Data

- `wardrobeItems` model supports draft/upload lifecycle: ready.
- Wardrobe setup profile is separate from item records: ready. Setup data is in `wardrobeSetupProfiles/{userId}`.
- `uploadStatus` lifecycle is defined: ready. `draft`, `upload_pending`, `uploaded`, `upload_failed`, and `deleted`.
- `analysisStatus` lifecycle is defined: ready. `not_requested`, `blocked_missing_consent`, `pending`, `completed`, and `failed`.
- `userId`/`ownerId` consistency is enforced: ready in shared helpers, Firestore rules, and backend finalisation.
- `storagePath` is generated by shared helpers: ready via `storagePaths.wardrobeOriginal` and wardrobe upload metadata helpers.
- Client cannot set backend-owned lifecycle fields after create: ready. Firestore rules deny client changes to upload and analysis lifecycle fields.

### C. Storage

- Upload path is private and exact: ready. `users/{userId}/wardrobe/{itemId}/original`.
- MIME type policy exists: ready. JPEG, PNG, WebP, HEIC, and HEIF.
- Max file size policy exists: ready. 10 MiB for wardrobe originals.
- Required metadata policy exists: ready. Owner, user, item, category, consent version, and exact storage path.
- Storage rules block public access: ready.
- Storage rules block cross-user upload: ready.
- Generated output paths are not client-writable: ready.
- Storage trigger QA passed: ready. Handler registration and lifecycle behavior are covered in emulators.

### D. Backend

- Storage finalisation handler exists: ready.
- Finalisation verifies metadata/path/owner/content type/size: ready.
- Valid upload can mark item uploaded: ready.
- Failure paths are safe: ready. Existing matching records can be marked `upload_failed`; missing records are audited without creating partial items.
- No AI job is created on upload: ready.
- Idempotency is handled: ready. Duplicate finalize leaves uploaded state stable.
- Orphan detection is non-destructive and documented: ready.

### E. Privacy

- Wardrobe photo upload is private: ready.
- Wardrobe photo analysis consent is separate from upload: ready.
- Missing analysis consent blocks analysis, not private upload: ready.
- Settings/privacy consent still works: ready.
- Deletion processor includes wardrobe-related records and Storage paths: ready.
- No public sharing exists: ready. `shared-with-stylist` exists only as a future type value; client-writable rules allow only `private`.
